# V3 Phase 1 (Foundation) -- 实现计划

> 日期: 2026-05-25
> 状态: 待上板验证

## 概述

V3p1 是 de2os 版本的基线验证阶段。所有核心组件（bootloader 上传、FreeRTOS 4 任务调度、CLI 19 命令、VGA 80x30 文本终端、PS/2 键盘输入）的代码已编写完成并通过交叉编译验证（73500 bytes @ 0x01000000），部署脚本已就绪。本阶段的目标是将这些组件在 DE2-115 板上逐一验证，确认从 FPGA 上电到 FreeRTOS shell 可交互的完整链路稳定可靠，为后续 Phase 2（像素模式 GUI）和 Phase 3（硬件引擎）打下基础。

## 实现步骤

### 步骤 1: Quartus 全量重编译

**目的**: 对齐最新的 RTL 代码和恢复的 SDC 约束文件，生成可信的 bitstream。

**操作**:
1. 确认 `par/constraints/de2extra.sdc` 存在且内容完整（含 CLOCK_50 20ns 基准时钟 + derive_pll_clocks）
2. 运行 `./run/deploy_de2shell_rtos.sh fpga` 或手动在 Quartus 中打开 `par/de2os/de2os.qpf` 执行全量编译
3. 检查编译结果：
   - 0 error, 0 critical warning
   - `de2os.sta.rpt` 中 `SDC File List` 包含 `../constraints/de2extra.sdc`
   - `Clocks` 表包含 `clk_50m` 和 PLL 派生时钟（不应只有 `altera_reserved_tck`）
   - `CLOCK_50` 不再显示为 `Unconstrained`

**预期结果**: `par/de2os/de2os.sof` 生成，STA 报告覆盖主时钟路径。

**排障**:
- 如果 SDC 仍然未被读取，检查 `de2os.qsf` 中的 `SDC_FILE` 路径是否正确指向 `../constraints/de2extra.sdc`
- 如果 `DRAM_CLK` 报 PLL 重复约束 warning，暂时可接受，不影响启动功能验证
- 资源不足：确认 `NUM_PARALLEL_PROCESSORS = 1`，检查是否因新增外设（PONG/Conway/BuildInfo）导致 LE 溢出

### 步骤 2: FPGA 烧录 + Bootloader 上传验证

**目的**: 验证完整的 boot chain -- FPGA 配置 -> bootloader 启动 -> UART 上传 -> SDRAM 执行。

**操作**:
1. 将 DE2-115 接通电源，USB-Blaster 连接 PC
2. 使用 `./run/deploy_de2shell_rtos.sh fpga` 烧录 `de2os.sof`
3. 打开串口终端（COM10, 115200 8N1），等待 bootloader 输出
4. 使用 `./run/deploy_de2shell_rtos.sh upload` 执行 UART 上传
5. 观察 UART 输出是否包含：
   ```
   === de2shell_rtos: FreeRTOS on NEORV32 ===
   HW build: ...
   FW: de2shell_rtos ...
   Starting FreeRTOS scheduler...
   ```

**预期结果**:
- Bootloader 正常输出 banner 并接受 'u' 命令进入上传模式
- 上传完成后 checksum 校验通过并显示 "OK"
- 发送 'e' 命令后固件从 0x01000000 开始执行，UART 输出启动信息
- FreeRTOS scheduler 成功启动

**排障**:
- 无 bootloader 输出: 确认烧录的是 `par/de2os/de2os.sof`（而非 `par/de2extra.sof`）；检查 KEY0 按键复位
- 上传失败: 检查 BIN 文件大小和 COM 端口号；`upload_de2os.py` 支持 115200/19200 双波特率自动检测
- 上传成功但无后续输出: 参考 `doc/phases/de2os-debug.md` 中的排查步骤；当前基线 `ICACHE_EN=false` 应避免此问题
- 如果出现 `crt0 panic`（死循环无输出），检查 GPIO LEDR 输出的 debug code：
  - `0xD010` = main 入口前崩溃
  - `0xD020` = 队列创建失败
  - `0xD030` = CLI 注册失败
  - `0xD040` = 任务创建失败
  - `0xDE10` = 应用中断
  - `0xDEE0` = 异常
  - `0xDEF0` = malloc 失败
  - `0xDED0` = 栈溢出

**预期结果**:
- 80 列 x 30 行文本网格清晰可见，无错位/乱码
- 颜色正确（青色标题、黄色信息、绿色提示符、白色输入）

- 颜色异常: 检查 RGB565 到 8-8-8 扩展的位移是否正确（R[15:11]<<3, G[10:5]<<2, B[4:0]<<3）

**目的**: 验证系统在 SDRAM 执行环境下长时间运行不崩溃。

**操作**:
1. 烧录 + 上传后，让系统运行至少 30 分钟
2. 期间交替执行 `info`、`hello`、`memtest`、`crypto` 等命令
3. 观察 LEDR 心跳灯（由 `vApplicationIdleHook` 驱动，低 4 位循环递增）
4. 检查 `stats` 和 `heapstat` 输出是否有异常变化

**预期结果**:

**排障**:

- LEDR 停止变化: 说明 idle hook 不再执行，可能某任务死循环或死锁

## 已知问题与风险

### 1. ICACHE 不可用（性能降级）

**来源**: `doc/phases/de2os-debug.md`

当前 `de2os_top.vhd` 中 `ICACHE_EN = false` 是已确认的稳定基线。开启 ICACHE 后，CPU 从 SDRAM 执行会因 ICACHE miss 触发连续 locked single-read 序列，暴露 `sdram_ctrl` 在连续请求下的 CDC 残余风险（`req_shadow` 覆盖窗口）。

**影响**: 关闭 ICACHE 导致每条指令都需要一次 SDRAM 读取，系统性能显著下降。对功能验证无影响，但 FreeRTOS 任务切换延迟和程序响应速度会比 V2 慢。

**缓解**: V3p1 阶段以功能验证为主，性能不作为阻塞项。后续 Phase 2+ 再集中修复 SDRAM CDC 并重新开启 ICACHE。

### 2. SDRAM 时序约束不完整

**来源**: `doc/phases/de2os-timing-status.md`

`par/constraints/de2extra.sdc` 已恢复，但仍缺：
- `DRAM_DQ[*]` 的 `set_input_delay`（读返回时序未建模）
- `DRAM_CLK` 被 `derive_pll_clocks` 和显式 `create_clock` 双重约束，可能产生 warning
- SDRAM I/O 的完整板级时序模型尚未建立

**影响**: 当前 STA 无法完整覆盖 SDRAM 读路径的 setup/hold。实际上 SDRAM 控制器已通过 memtest 验证（4096-word dense + 31 sparse boundary），但缺少 sign-off 级别约束。

**缓解**: V3p1 以实测结果为准。如果板上 SDRAM 读写出错，优先查 CDC 和状态机逻辑，再考虑补充 SDC 约束。

### 3. FreeRTOS heap 在 SDRAM，DMEM 布局紧凑

**来源**: `doc/phases/de2os-rtos-status.md`

FreeRTOS heap 已移至 SDRAM 0x01900000（16384 bytes），framebuffer 在 0x01800000。DMEM 仅剩 16KB 供 `.data`、`.bss`、内核小对象和主栈使用。当前 `.bss = 32640 bytes`，但其中大部分（`ucHeap` 16384 bytes）已通过 `rtos_memory.c` 链接到 SDRAM。

**影响**: 如果新增全局变量导致 `.data + .bss` 超过 DMEM 容量，链接器会报错。

**缓解**: 使用 `riscv-none-elf-size` 定期检查 memory utilization。大块数据必须显式放到 SDRAM section。

### 4. deploy_de2shell_rtos.sh 固件路径

**来源**: `run/deploy_de2shell_rtos.sh`

`BIN` 路径指向 `sw/app/de2shell_rtos/neorv32_exe.bin`，与目标固件一致。

**已解决**: 旧的 `deploy_de2os.sh`（指向 `sw/app/de2os/`）已删除。统一使用 `deploy_de2shell_rtos.sh`。

### 5. PS/2 仅支持 ASCII 路径

**来源**: `doc/phases/de2os-rtos-status.md`

当前 `t_uart_input` 只将可解码为 ASCII 的 PS/2 按键送入输入队列。方向键、功能键、组合键不传递。

**影响**: shell 交互足够，但 GUI 程序（Win30）的键盘体验不完整。

**缓解**: V3p1 只验证基本 PS/2 输入（ASCII 字符）。richer event 通道留到后续阶段。

### 6. PONG/Conway 引擎已 park

**来源**: `src/rtl/de2os_top.vhd` 第 743-757 行

PONG 和 Conway 硬件引擎在 `de2os_top` 中被 park（`wb_dat_i <= (others=>'0'); wb_ack <= wb_stb;`），不会参与正常总线响应。

**影响**: 不影响 V3p1 验证。但 `startui` 命令如果依赖 Conway HW 引擎，会收到全零响应。

**缓解**: V3p1 的 `startui` 测试以软件渲染为主，硬件引擎验证推迟。
