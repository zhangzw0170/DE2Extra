# 编译烧录前必看

## 当前结论

现在这套工程“无法正常编译烧录”的首要原因，不在 JTAG，也不在板卡本身，而是在 `build.sh` 的固件构建链路。

> 2026-05-24 补充：即便按本文前述要求切到 Git Bash，`build.sh app/de2shell` 这层包装在当前机器上仍可能出现**不稳定伪失败**。  
> 已验证稳定的路径是：
> 1. 手动 `docker run ... make clean && mkdir -p build && make image`
> 2. 手动复制 `sw/app/de2shell/neorv32_imem_image.vhd` 到 `src/rtl/neorv32_imem_image.vhd`
> 3. 手动运行 `quartus_sh --flow compile`
>
> 也就是说，**Docker 编译器、Quartus、本地工程本身都能正常工作，但 `build.sh` 这个总包装脚本暂时不能视为唯一稳定入口。**

最先出现的硬错误是：

```text
conway_ed.c:144:1: fatal error: opening dependency file build/conway_ed.c.d: No such file or directory
```

这说明流程在 Quartus 和烧录之前，就已经死在固件编译阶段了。

## 根因 1：`build.sh` 里的 `make` 调用方式有问题

旧脚本使用的是：

```bash
make clean all image NEORV32_HOME=/project/neorv32
```

这条命令把 `clean` 和构建目标放进了同一次 `make`。

而 NEORV32 的 `common.mk` 里，`build/` 目录是通过 order-only prerequisite 自动创建的。问题在于：

1. `make` 启动时如果看到 `build/` 已存在，会认为不需要再建。
2. `clean` 在执行过程中把 `build/` 删掉。
3. 后面的编译阶段不会再次触发 `mkdir -p build`。
4. 编译器开始写 `build/*.d` 依赖文件时，目录已经没了，于是直接失败。

正确做法是分步执行：

```bash
make clean NEORV32_HOME=/project/neorv32
mkdir -p build
make image NEORV32_HOME=/project/neorv32
```

但要注意：这只能解释最早的 `build/*.d` 目录缺失问题，**不能保证 `build.sh` 包装层在当前机器上完全稳定**。

## 根因 2：这条脚本当前必须从 Git Bash 运行

在 Windows 上，`build.sh` 当前按 Git Bash / MSYS 环境编写。

推荐用法：

```bash
MSYS_NO_PATHCONV=1 bash build.sh --flash app/de2shell
```

如果你人在 PowerShell 里，又想明确调用 Git Bash，可直接运行：

```powershell
& 'E:\Software\Scoop\apps\git\current\bin\bash.exe' -lc "cd /e/Main/JuniorII/NonExam/FPGA/DE2Extra && MSYS_NO_PATHCONV=1 bash build.sh --flash app/de2shell"
```

不要直接在 PowerShell 里执行：

```powershell
bash build.sh --flash app/de2shell
```

当前机器上这样做通常会落到 WSL 的 `bash`，然后报：

```text
execvpe(/bin/bash) failed: No such file or directory
```

这不是项目代码错误，而是运行环境选错了。

## 根因 2.5：`build.sh` 包装层当前仍可能出现“同命令手动能过、脚本里却失败”的情况

当前实测出现过这样一种现象：

1. 用 Git Bash 运行 `build.sh app/de2shell`
2. Docker 阶段报链接错误，例如：

```text
ld: cannot find build/main.c.o
ld: cannot find build/memtest.c.o
...
```

3. 但把 `build.sh` 里的 Docker 命令原样拆开，手动执行：

```bash
docker run --rm -v 'E:\Main\JuniorII\NonExam\FPGA\DE2Extra:/project' de2extra-builder \
  bash -lc 'export PATH=/opt/riscv/bin:$PATH; \
            cd /project/sw/app/de2shell; \
            make clean NEORV32_HOME=/project/neorv32; \
            mkdir -p build; \
            make V=1 image NEORV32_HOME=/project/neorv32'
```

却可以稳定成功。

这说明当前的真实情况是：

- `make clean`
- `mkdir -p build`
- `make image`
- `/opt/riscv/bin/riscv-none-elf-gcc`

这几步本身都没坏；

坏的是**外层包装调用链的稳定性**。

因此，后续如果再遇到 `build.sh` 阶段性失败，不要第一时间怀疑 RTL 或 C 代码本身，应优先切换到“手动 Docker 构建”路径。

## 根因 3：Docker 镜像里的 RISC-V 工具链不在默认 `PATH`

`de2extra-builder` 镜像内实际存在交叉编译器：

```text
/opt/riscv/bin/riscv-none-elf-gcc
```

但默认 `PATH` 不包含 `/opt/riscv/bin`。

因此脚本里需要显式加入：

```bash
export PATH=/opt/riscv/bin:$PATH
```

否则后续很容易在编译器查找阶段再次失败。

## 根因 4：`de2shell` 已超过 32KB IMEM，硬件必须使用 64KB

`de2shell` 当前生成的 IMEM 镜像大小约为：

```text
39056 bytes
```

这已经超过 32KB，因此如果顶层硬件还保持：

```vhdl
IMEM_SIZE => 32*1024
```

Quartus 后续会因为 IMEM 溢出失败。

当前工程已经改成：

```vhdl
IMEM_SIZE => 64*1024
```

所以这项目前不是第一阻塞点，但后续如果有人把顶层又改回 32KB，会重新出问题。

## 关于 Zfinx FPU

当前项目目标以整数运算、密码扩展和有限域计算为主，不再启用 Zfinx FPU。

需要保持三处一致：

1. 硬件 generic：`RISCV_ISA_Zfinx => false`
2. 软件编译参数：应用 `MARCH` 不再带 `_zfinx`
3. 屏幕 / 串口提示字符串：不要再宣称支持 `Zfinx`

如果只改其中一处，后面很容易出现“硬件配置、软件 ISA、界面提示三者不一致”的假象。

## 烧录前检查清单

1. 从项目目录打开 Git Bash。
2. 确认 Docker、Quartus 23.1std Lite、USB-Blaster 已就绪。
3. 确认板卡上电，JTAG 可见。
4. 确认没有残留的 `quartus_*` 或旧下载进程占用流程。
5. 确认顶层 IMEM 仍为 64KB。
6. 运行：

```bash
MSYS_NO_PATHCONV=1 bash build.sh --flash app/de2shell
```

如果这条命令再次出现“不该有的” Docker 链接失败，请直接改走下面这套当前稳定路径。

## 最容易重复踩坑的一点：改了代码，不等于板上已经是新镜像

这项目当前最常见的误判，是把“源码已经改对”误认为“FPGA 已经跑上新版本”。

实际必须走完整链路：

1. 重新生成 `sw/app/de2shell/neorv32_imem_image.vhd`
2. 复制到 `src/rtl/neorv32_imem_image.vhd`
3. 重新跑 Quartus，生成新的 `par/de2extra.sof`
4. 重新 `quartus_pgm` 烧录

只做前两步，不会自动让板上变成新版本；只重新烧旧的 `par/de2extra.sof`，行为也不会变。

如果你怀疑“这次修改怎么完全没生效”，先不要急着继续改代码，先查这三件事：

1. `src/rtl/neorv32_imem_image.vhd` 是否真的是刚拷过去的新文件
2. `par/de2extra.sof` 是否是在这次复制之后重新编出来的
3. 板上复位后 `dashboard` 的 `Uptime` 是否从很小的值重新开始计数

如果 `Uptime` 还是上一次的延续值，或者板上行为还是旧版本特征，就优先怀疑“这次根本没烧上去”，不要先怀疑业务逻辑。

## IR 验证口径也要固定，不要再按旧 CH 遥控器理解

当前这只遥控器必须以 `FPGA/Exp/Exp10/Exp10实验指导书原文.md` 的表 6-16 为准。

最容易误判的一条是：

```text
0x1A = CH+
```

这里的坑在于：表 6-16 里的 `0..23` 是**按物理位置顺序编号**，不是遥控器上印出来的数字标签。

因此 `de2shell` 当前的 IR 验证口径应固定为：

- `1` -> hello
- `2` -> memtest
- `3` -> crypto
- `4` -> snake
- `5` -> life
- `6` -> dashboard
- `7` -> info
- `0` / `RETURN` -> return shell
- `CH-` -> previous program
- `CH+` -> next program

如果板上现象和这套口径不一致，先确认：

1. 你现在跑的是不是刚烧进去的新 `sof`
2. 对照的是不是 Exp10 表 6-16，而不是另一只遥控器的 CH 码表

如果只是复用已有的 `par/de2extra.sof`，跳过 Docker 固件编译与 Quartus 全流程，可运行：

```bash
MSYS_NO_PATHCONV=1 bash build.sh --flash-only
```

## de2os 特例

`de2os` 不是走上面的通用 `build.sh` 流程，而是走专用脚本：

```bash
./run/deploy_de2os.sh all
```

原因有两点：

1. `de2os` 使用独立 Quartus 工程 `par/de2os/de2os.qpf`
2. `de2os` 固件位于 `sw/app/de2os/`，并通过 bootloader 上传到 `0x01000000`

### 当前稳定基线

截至 2026-05-24，`de2os` 当前已验证可启动的硬件基线是：

```vhdl
BOOT_MODE => 0
ICACHE_EN => false
```

这不是最终性能配置，只是当前用于继续软件修复的稳定配置。若后续重新打开 `ICACHE`，应视为单独的性能/硬件问题排查，而不是默认安全改动。
