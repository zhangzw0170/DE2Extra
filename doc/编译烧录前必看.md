# 编译烧录前必看

## 先看结论

现在这套工程默认走的是 `bootloader + UART upload`（boot mode 0）。

现在主张的是：`增量更新优先`。

### Boot Mode 0 vs Boot Mode 2

| | Boot Mode 0（推荐） | Boot Mode 2（仅调试用） |
|---|---|---|
| IMEM 内容 | ~2KB bootloader | 完整程序 (~70KB) |
| 程序存储 | SDRAM 0x01000000 | IMEM (FPGA bitstream 内) |
| 上传方式 | UART 串口上传 | 无需上传，上电即运行 |
| 脚本 | `deploy_de2shell_rtos.sh` | `deploy_de2shell_rtos_imem.sh` |
| 改代码后 | 只需 `app` 或 `upload` | 必须完整 `full` 重编 |
| 耗时 | ~48s (app) 或 ~5s (upload) | ~4min (full) |

**结论**：日常开发用 boot mode 0。boot mode 2 仅用于初始调试或无法使用串口的场景。

这意味着：

1. 改 `sw/app/de2os/` 或 `sw/app/de2shell_rtos/`，大部分情况下只需要重新编译 app，然后串口上传。
2. 如果 `bin` 已经编好，连重新编译都不需要，直接上传即可。
3. 不要再把“改了软件”默认等同于“必须重新跑 Quartus”。
4. 只有改到 `RTL / 顶层 / bootloader / IMEM 配置 / 地址映射`，才需要重新综合烧录 FPGA。

## 增量更新分层

以后默认按下面四档理解，不要一上来就跑整包：

### 第 1 档：`upload`

只上传当前现成的 `neorv32_exe.bin`。

适用：

- 代码没变，只想重新下发一次
- 已经在别处编好，不想本机再编
- 只是在板子上验证串口下载链路

### 第 2 档：`app`

重新编译 app，然后只上传 app。

适用：

- 改了 CLI
- 改了 FreeRTOS 任务
- 改了 GUI 软件逻辑
- 改了 `fb_hal.c` / `vga_hal.c` / 普通 C 代码

这是后续最常用的日常调试档。

### 第 3 档：`fpga`

只重烧当前 `.sof`，不重新上传 app。

适用：

- 你确认 bitstream 变了，但 app bin 不需要重新生成
- 只是想把已经生成好的 FPGA 镜像重新打进板子

### 第 4 档：`full`

重新编译 app，必要时重编 bootloader，重新综合并烧录 FPGA，最后再上传 app。

适用：

- 改了 `src/rtl/`
- 改了 bootloader
- 改了 IMEM/DMEM 配置
- 改了地址映射
- 改了必须固化进 bitstream 的内容

结论：平时优先用 `upload` 或 `app`，`full` 只在确实跨到硬件层时才用。

## 运行环境

推荐从 Git Bash 运行部署脚本：

```powershell
& 'E:\Software\Scoop\apps\git\current\bin\bash.exe' run/deploy_de2shell_rtos.sh app
```

IMEM 直接烧录版本（boot mode 2，仅用于初始调试）：

```powershell
& 'E:\Software\Scoop\apps\git\current\bin\bash.exe' run/deploy_de2shell_rtos_imem.sh all
```

不要直接在 PowerShell 里写：

```powershell
bash run/deploy_de2shell_rtos.sh app
```

这台机器上的 `bash` 默认会落到 WSL，并报：

```text
execvpe(/bin/bash) failed: No such file or directory
```

这不是工程问题，是跑错了 shell。

## de2shell_rtos

目标：

- 工程目录：`sw/app/de2shell_rtos/`
- Quartus 工程：`par/de2os/de2os.qpf`
- 上传目标：`0x01000000`
- 入口脚本：`run/deploy_de2shell_rtos.sh`

### 正常软件迭代

只改了 `de2shell_rtos` 本身、CLI、FreeRTOS 任务、GUI、VGA/FB 软件层、普通 C 代码时，用：

```bash
./run/deploy_de2shell_rtos.sh app
```

这条命令会做两件事：

1. 重新编译 `sw/app/de2shell_rtos/neorv32_exe.bin`
2. 通过 bootloader 以 `115200` 串口上传

如果板子已经在跑应用，脚本会等待 bootloader，此时按一下 `KEY0` 即可。

### 仅上传当前 bin

如果你确认 `sw/app/de2shell_rtos/neorv32_exe.bin` 已经是最新的，不想再编译，可以用：

```bash
./run/deploy_de2shell_rtos.sh upload
```

### 什么时候必须重综合

以下改动必须用：

```bash
./run/deploy_de2shell_rtos.sh full
```

或至少：

```bash
./run/deploy_de2shell_rtos.sh fpga
```

典型触发条件：

- 改了 `src/rtl/` 下的 RTL
- 改了 `src/rtl/de2os_top.vhd`
- 改了 `src/rtl/periph/vga_text_terminal.vhd`
- 改了 bootloader
- 改了 NEORV32 IMEM/DMEM 配置
- 改了总线地址映射

### 本次实测耗时

`2026-05-25` 这次 `full` 的实测时间：

- app 编译：`25s`
- bootloader 重编：`17s`
- Quartus 综合：`152s`
- JTAG 烧录：`12s`
- UART 上传：`23s`

合计约 `229s`，也就是接近 `4` 分钟。

因此正常软件调试时，尽量走 `upload` 或 `app`，不要动不动就跑 `full`。

## de2os

目标：

- 工程目录：`sw/app/de2os/`
- Quartus 工程：`par/de2os/de2os.qpf`
- 上传目标：`0x01000000`
- 入口脚本：`run/deploy_de2shell_rtos.sh`

### 正常软件迭代

只改了 `sw/app/de2os/` 的软件逻辑时，用：

```bash
./run/deploy_de2shell_rtos.sh app
```

这条命令会：

1. 重新编译 `sw/app/de2os/neorv32_exe.bin`
2. 等待 bootloader
3. 通过 UART 上传

如果板子当前正在跑应用，按一下 `KEY0` 即可进入 bootloader。

### 仅上传当前 bin

如果你确认 `neorv32_exe.bin` 已经是最新的，不想再编译，可以用：

```bash
./run/deploy_de2shell_rtos.sh upload
```

### 什么时候必须重综合

只有这些情况才需要：

```bash
./run/deploy_de2shell_rtos.sh full
```

或：

```bash
./run/deploy_de2shell_rtos.sh fpga
```

典型触发条件：

- 改了 `src/rtl/`
- 改了 `par/de2os/` 相关顶层配置
- 改了 bootloader
- 改了片上存储大小
- 改了外设地址映射
- 改了必须固化到 bitstream 里的内容

结论很简单：`de2os` 现在在大部分情况下也不需要重新综合。

## “烧录”和“下载”分开理解

这里后面统一用两种词：

- `烧录`：JTAG 把 `.sof` 打进 FPGA
- `下载`：bootloader 通过 UART 把 `neorv32_exe.bin` 发进应用地址

所以：

1. 平时软件迭代，大多数只是“下载”，不是“重新烧录”
2. 只有硬件侧变了，才需要“烧录 + 下载”

## KEY0 和“为什么屏幕没变”

`KEY0` 是板级复位。

当前流程里，按下 `KEY0` 后会先回到 NEORV32 bootloader；而 bootloader 默认只管 UART，不负责清屏或重画 VGA。

所以会出现这种现象：

1. 你已经按了 `KEY0`
2. 串口上 bootloader 确实出来了
3. VGA 还是停留在上一帧内容

这不代表 reset 失效，也不代表新固件没烧上去；只是 bootloader 没接管 VGA。

## 最常见误判

### 1. 改了源码，但板上没变

先区分你改的是哪一层：

- 改软件：优先 `upload`，其次 `app`
- 改 RTL：重新跑 Quartus + JTAG 烧录，之后再上传 app

### 2. 只重新按了 `KEY0`，以为已经换成新固件

不是。

`KEY0` 只会复位并回到当前 bitstream 里的 bootloader，不会自动把你工作区里的新 `bin` 下载进板子。

### 3. 误把“VGA 画面没清”当成“没复位”

也不是。

bootloader 不清 VGA，这只是显示残留。

## 历史坑

旧文档里提到的这些问题，今天仍然值得记住，但不再是主流程：

- `make clean` 和 `make all/image` 混在一次调用里，会让 `build/` 被删掉后不重建
- Docker 里 RISC-V 工具链需要显式走 `/opt/riscv/bin/riscv-none-elf-`
- 某些老脚本是 IMEM 固化流，不适合现在这套 bootloader-first 流程

如果以后又遇到“明明只改了 app，却被迫重新综合”的情况，优先先查脚本和流程，不要先怀疑业务代码。
